# OJ 题库抽样检查报告

- 检查时间: 2026-04-22 19:50:18 CST
- 检查对象: `cdut-oj-backend` 中的题库数据与前台题目详情页
- 题库总量: 609 题
- 检查方式: 数据层统计 + 抽样题目详情页人工核查 + API 返回验证

## 一、执行摘要

本次对 OJ 题库进行了抽样与结构性检查，结论是：

1. 题库存在明显的批量数据质量问题，不是个别题目问题。
2. 问题主要集中在「结构化字段缺失/失真」「题目唯一标识冲突」「前台展示质量不一致」三个层面。
3. 其中最严重的问题是重复 `_id` 导致题目详情接口直接报 `server error`，用户无法打开对应题目。
4. 其次是大量题目把输入输出样例写进 `description`，但 `input_description`、`output_description`、`samples` 等结构化字段为空或无效，导致前台页面出现空白分区、样例缺失、内容错位。
5. 另外还存在标题空白字符污染、来源缺失、难度标签单一化、测试点评分元信息缺失等系统性问题，说明题库导入过程缺少统一清洗、校验和回归检查。

## 二、总体统计发现

基于数据库统计得到以下问题规模：

- 总题目数: 609
- 重复 `_id` 的题目标识数: 3 组
- `input_description` 为空: 113 题
- `output_description` 为空: 113 题
- `samples` 存在结构问题或内容缺失: 112 题
- 标题存在前后空格: 1 题
- 标题含 NBSP / 全角空格等异常空白字符: 36 题
- `source` 为空: 1 题
- `description` 内嵌“输入/输出/样例”段落，未拆到结构化字段: 52 题
- `test_case_score` 为空: 607 题
- `test_case_score` 全为 0: 2 题
- `difficulty` 分布: 609 题全部为 `Mid`

说明：
- 607 题 `test_case_score=[]` 未必立即导致 ACM 判题错误，但说明题目元信息并不完整，后续若切换 OI 规则、统计逻辑或导出工具时会产生兼容风险。
- 609 题全部标记为 `Mid`，说明难度字段基本失去区分度，难以用于筛题、教学分层和推荐。

## 三、抽样题目与发现

本次重点抽查了以下类型的样本：

1. 正常可访问但内容质量有问题的题目
2. 样例结构缺失的题目
3. 标题存在异常空白字符的题目
4. `_id` 冲突导致详情页报错的题目

### 1. 样本 A: `fps-7681` - 送分题-A+B Problem

现象：
- 前台可正常打开。
- 题面展示中出现异常 UI 文本混入描述内容：`Calculate a+b Button`
- 浏览器 DOM 中存在链接文本 `Show`，被用于不该出现的位置。
- `source` 为“系统原理，熟悉OJ”，更像教学备注而非标准题目来源。
- `test_case_score` 虽存在，但 5 个测试点 `score` 全为 0。

证据：
- API `GET /api/problem?problem_id=fps-7681` 返回：
  - `description = "Calculate $ a+b $ <span class=\"ui small primary button\">Button</span> <br />"`
  - `samples = [{"input":"1 2","output":"3"}]`
  - `test_case_score` 全部为 0
- 前台浏览器详情页可见描述中出现 `Button` 字样。

问题判断：
- HTML 内容没有经过统一清洗，编辑器/按钮样式片段被带入题面。
- 结构可用，但内容质量不达标。

建议：
- 清洗 `description` 中的展示型 HTML，移除无业务语义的 `<span class="ui ... button">Button</span>`。
- 对题面允许的 HTML 标签做白名单，仅保留 `p / pre / code / ul / ol / li / strong / em / img / math` 等必要标签。
- 对 `test_case_score` 增加生成规则：ACM 题可按测试点数量生成默认占位分值或显式标记“非 OI 计分”。

### 2. 样本 B: `fps-e33e` - 怪脾气的评测系统

现象：
- 前台详情页中 `Description` 有内容，但 `Input` / `Output` 区域为空。
- `Sample Input 1` / `Sample Output 1` 只有标题，没有实际样例。
- 数据库存储中 `samples=[{"input": null, "output": null}]`
- `input_description` 与 `output_description` 都为空字符串。
- `hint` 存在，但呈现方式比较原始。

证据：
- API/数据库数据：
  - `input_description = ""`
  - `output_description = ""`
  - `samples = [{"input": null, "output": null}]`
- 前台页面可见空白的 Input/Output 区域和空白样例块。

问题判断：
- 题目正文有内容，但结构化字段未被正确填充。
- 导入过程未把“题目正文中的输入输出信息”拆解进专用字段。
- 前端没有对空样例做兜底隐藏，导致用户看到空壳区块。

建议：
- 编写一次性修复脚本：
  - 把 `samples` 中 `input/output` 为 `null` 的记录视为无效样例并清洗。
  - 对 `description` 中可识别的输入/输出段落进行抽取并回填。
- 前端增加保护逻辑：
  - 若 `input_description` 为空则隐藏 Input 区块或提示“未提供独立输入说明”。
  - 若样例为空或 `null`，不渲染 `Sample Input/Output` 标题。

### 3. 样本 C: `fps-988b` - C语言-水仙花数

现象：
- 前台可打开，但题面中把输出样例残片直接混入 `Description`。
- 描述中可见 `Output:` 和 `153 ??? ??? ???`，明显是导入污染或样例缺失占位未清理。
- `samples=[{"input": null, "output": null}]`
- `source` 为空。
- `test_case_score` 存在，但全部为 0。

证据：
- 数据库存储：
  - `description_preview` 包含 `Output:` 与 `???`
  - `samples` 为 null 样例
  - `source = null`
- 前台页面直接显示 `153 ??? ??? ???`

问题判断：
- 题目描述、输出说明、样例之间边界混乱。
- 这是典型的“从旧题库/富文本导入到结构化题库时未拆字段”的问题。

建议：
- 清洗含 `???`、`Output:` 等明显占位或混排文本的题目。
- 建立“导入后自动体检规则”，识别以下风险文本：
  - `???`
  - `Output:` / `Input:` 混入 description
  - `samples` 为 `null`
- 对 `source` 为空的题目补录来源，若无法确认则统一标记为“待补充来源”。

### 4. 样本 D: `fps-5d17` - 走楼梯的方法

现象：
- 页面展示标题为“走楼梯的方法”，但数据库原始标题有前导空格。
- `description` 中直接嵌入了“【输入文件】【输出文件】【输入样例】【输出样例】”。
- `input_description` 与 `output_description` 为空。
- `samples` 为 `null` 样例，前台样例区块为空。

证据：
- 数据库标题：`" 走楼梯的方法"`
- `description` 中包含输入/输出/样例完整说明
- 前台 `Input` / `Output` / 样例区域为空壳

问题判断：
- 标题被前端 `trim` 了，但底层数据仍不规范。
- 结构化抽取缺失，导致本来信息完整的题目在页面上表现反而不完整。

建议：
- 对所有标题做标准化清洗：`strip()` + 替换 NBSP/全角空格。
- 对 description 中嵌套的标准段落进行批量解析拆分：
  - `【问题描述】` -> description
  - `【输入文件】` -> input_description
  - `【输出文件】` -> output_description
  - `【输入样例】/【输出样例】` -> samples

### 5. 样本 E: `fps-615d` - 练65.3 螺旋矩阵

现象：
- 可正常打开，结构化字段较完整。
- 但标题原始数据中含异常空白字符（NBSP 或类似空格污染）。
- 前台虽然显示正常，但底层数据不干净，说明存在批量导入格式污染。

证据：
- 统计发现 36 题标题含 NBSP / 全角空格。
- 抽样该题命中这一类问题。

问题判断：
- 当前多数字体下“看起来正常”，但会影响：
  - 标题检索
  - 文本比对
  - 题目去重
  - 导出和外部同步

建议：
- 对标题和来源等短文本字段做 Unicode 空白标准化。
- 在导入/编辑保存时统一做 normalization。

### 6. 样本 F: `_id` 冲突题目（严重）

发现了至少 3 组重复 `_id`：

- `fps-7c61`
  - id 162: `【例8.1】人民币支付`
  - id 186: `练11.4计算(a+b)×c的值`
- `fps-15e5`
  - id 430: `【例51.1】 删除元素`
  - id 547: `【例72.3】 提取数字串按数值排序`
- `fps-a49c`
  - id 579: `【例82.2】 座次问题`
  - id 580: `【例82.3】 公平抽签`

现象：
- 前台访问 `/problem/fps-7c61`、`/problem/fps-a49c` 时直接显示 `server error`
- API `/api/problem?problem_id=fps-7c61` 返回：
  - `{"error":"server-error","data":"server error"}`
- 同类问题会导致题目详情页无法使用、提交区不可用、元信息空白。

问题判断：
- 前后端都依赖 `_id` 作为公开题目标识。
- 当 `_id` 不唯一时，后端按 `_id` 查询题目会触发多记录冲突，导致接口失败。
- 这是最高优先级问题，因为它直接影响访问可用性。

建议：
- 立即在数据库层为 `Problem._id` 增加唯一性约束（在完成数据修复后执行）。
- 先跑一次重复 `_id` 审计脚本，生成修复清单。
- 为冲突题目重分配新 `_id`，并同步更新所有引用关系。
- 导入流程中加入唯一性检查，禁止重复 `_id` 入库。

## 四、问题归类

### P0 / 高优先级

1. 题目 `_id` 重复，导致详情页与 API 直接报错
2. 题目样例为 `null`，前台仍渲染空样例区域
3. 输入/输出说明缺失，但内容其实埋在 description 中，前台表现为信息缺失

### P1 / 中高优先级

1. 题面 description 混入 UI/HTML 垃圾文本（如 `Button`）
2. description 中混入 `Output:`、`???` 等无效占位内容
3. 标题异常空白字符污染
4. 来源字段缺失或不规范

### P2 / 中优先级

1. 难度字段全部为 `Mid`，缺乏区分度
2. `test_case_score` 大面积为空，题目元信息完整性较差
3. 富文本格式不统一，代码块/矩阵/样例呈现质量一般

## 五、根因判断

综合判断，这些问题大概率来自以下几个根因：

1. 旧题库向当前 OJ 导入时，未完成“富文本 -> 结构化字段”的映射拆分。
2. 导入脚本缺少数据清洗：
   - 没有 trim / Unicode normalization
   - 没有过滤 `null sample`
   - 没有识别 `???` 等占位文本
3. 导入后缺少一致性校验：
   - `_id` 未做唯一性检查
   - `samples` 未校验必填性
   - `input_description/output_description` 与 description 未做互补校验
4. 前端展示层缺少兜底：
   - 空样例仍然渲染
   - 空输入输出区块仍然显示
   - API server-error 页面未提供明确原因

## 六、建议解决方案

### 方案 A：先做数据体检与批量修复（推荐先执行）

编写一次性审计与修复脚本，至少覆盖：

1. 重复 `_id` 检测与修复候选清单生成
2. 标题/来源字段清洗：
   - 去前后空格
   - 替换 NBSP / 全角空格
3. `samples` 清洗：
   - 删除 `{"input": null, "output": null}` 之类无效样例
4. description 结构抽取：
   - 自动识别 `【输入文件】`、`【输出文件】`、`【输入样例】`、`【输出样例】`
   - 拆分并回填到结构化字段
5. 风险内容识别：
   - `???`
   - `Output:` / `Input:` 残片
   - `Button` 等 UI 污染

### 方案 B：为导入流程建立强校验

以后新题导入前，必须通过以下校验：

- `_id` 唯一
- title 非空且标准化
- 至少有有效 description
- `samples` 中每条都必须有完整 input/output，或者允许明确为空但前端不展示
- `input_description` / `output_description` 若为空，应检查 description 中是否混入结构化段落
- source、difficulty、languages 做最小合法性校验

### 方案 C：前端做容错与降级显示

1. `samples` 无效时不显示 Sample 区块
2. `input_description` / `output_description` 为空时不显示空标题区块
3. 当 API 返回 `server-error` 时，展示明确提示：
   - 题目数据异常
   - 联系管理员修复
   - 给出问题编号
4. `source` 若为 URL，则渲染为可点击链接
5. 对 description 中的代码块、矩阵、样例文本增强样式支持

### 方案 D：建立持续质量监控

建议增加一个“题库体检脚本”并定期运行，输出 Markdown/CSV 报告，指标至少包括：

- 重复 `_id`
- 空/异常样例
- 空输入输出说明
- 标题空白污染
- 来源缺失
- 难度异常分布
- test_case_score 缺失

这样可以在每次批量导题后立即发现问题，而不是等到用户点开题目才暴露。

## 七、建议修复优先级与落地顺序

### 第一阶段（立即处理）

1. 修复重复 `_id` 题目
2. 修复 `samples = null` 的题目并隐藏空样例区块
3. 修复有完整描述但结构化字段为空的题目（优先抽取输入/输出/样例）

### 第二阶段（本周内处理）

1. 清洗标题异常空格和来源字段
2. 清洗 description 中的 `???`、`Button` 等污染文本
3. 对高频访问题目做人工复核

### 第三阶段（体系化治理）

1. 为 `_id` 加唯一约束
2. 建立导入校验和定期体检脚本
3. 建立题库质量评分机制
4. 重新梳理 difficulty 字段，使其具备真实教学意义

## 八、最终结论

当前题库“能用”，但“质量不稳定”。

对普通用户而言，最直接的问题有三类：

1. 有些题目点开直接报错（重复 `_id`）
2. 有些题目输入输出和样例区域是空的（结构化字段缺失）
3. 有些题目文本本身存在污染、格式混乱或样例残缺（导入质量问题）

建议优先把这次检查识别出的三类核心问题做成批量修复任务：

- 任务 1：重复 `_id` 修复
- 任务 2：空样例/空输入输出结构修复
- 任务 3：题面文本清洗与标准化

完成这三项后，题库的可访问性、可读性和维护性会显著提升。
