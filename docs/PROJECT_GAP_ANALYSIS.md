## 二、功能性缺失（按模块分类）

### 2.1 竞赛(Contest)模块 — P0 严重缺失

**现状**: 0个竞赛，OJ后端支持Contest但从未创建过

**需要开发**:
- [ ] **管理员创建竞赛页面/流程**: OJ Admin后台可创建Contest，但需配置题目集、时间窗口、参赛规则
- [ ] **竞赛题目集**: 至少准备3-5套不同难度的竞赛题目（入门/进阶/模拟赛）
- [ ] **竞赛教练(AI)联动**: ContestCoachAgent需能读取当前竞赛上下文（题目、时间、排名）
- [ ] **竞赛模式前端**: 计时竞赛界面或至少在AI聊天中识别"正在参赛"状态
- [ ] **竞赛数据收集**: 参赛提交数据、排名变化、完成率统计

**验收标准**:
- 管理员可在OJ后台创建Contest并配置题目
- 学生可从AI聊天界面查看到当前/近期竞赛信息
- 至少有2场以上竞赛的实际使用数据

---

### 2.2 智能题目推荐系统 — P1

**现状**: 后端有 `submission_recommendation.py`，但无前端入口，无推荐算法

**需要开发**:
- [ ] **推荐算法**: 基于学生提交记录（AC/WA/TLE分布）+ 知识点标签，推荐合适难度的下一题
- [ ] **推荐API端点**: `/api/recommendation/next-problem?user_id=X`
- [ ] **前端推荐入口**: 解题完成后显示"推荐下一题"、侧边栏"为我推荐"按钮
- [ ] **推荐解释**: 给出推荐理由（"你在这个知识点上有3次WA，试试这道"）

**验收标准**:
- 学生做完一题后，系统推荐1-3道相关题目并附带理由
- 推荐考虑学生历史提交、知识点掌握度、难度递进

---

### 2.3 知识图谱 — P1

**现状**: 46个扁平标签，无层级关系、无边权、无推理

**需要开发**:
- [ ] **知识点层级建模**: 从扁平标签扩展为树状结构（如: 数据结构→树→二叉搜索树）
- [ ] **知识点关联数据库表**: `ai_agent.knowledge_nodes` + `ai_agent.knowledge_edges`
- [ ] **前置依赖关系**: 定义学习路径上的先修知识（如"学并查集前需会树和集合"）
- [ ] **前端知识图谱可视化**: 简单的节点图展示，标注学生当前掌握位置
- [ ] **知识点-题目映射**: 每题关联1-N个知识点，而非仅1个标签

**最低交付**:
- 至少3层深度的知识点树（总节点≥100）
- 学生在AI聊天中可查看自己知识点掌握概览
- 后端API可返回某知识点的题目列表

### 2.4 学习路径规划 — P1

**现状**: LearningManagerAgent依赖LLM临时生成，无持久化、无算法

**需要开发**:
- [ ] **用户学习画像表**: `ai_agent.user_learning_profile`（已有设计方案见 `docs/plans/student-learning-state-management-solution.md`）
- [ ] **学习事件流表**: `ai_agent.learning_events`
- [ ] **路径推荐算法**: 基于掌握度+知识点依赖图，生成个性化学习路径
- [ ] **持久化学习状态**: 跨会话累积学生的知识点掌握、效率趋势、挫败风险
- [ ] **前端进度面板**: 显示学习路径进度、当前阶段、推荐下一步

**最低交付**:
- 学生可看到自己的学习数据概览（已做题数、AC率、薄弱知识点）
- AI对话中LearningManagerAgent能引用学生历史数据做建议

### 2.5 代码审查结构化输出 — P1

**现状**: CodeReviewerAgent返回纯文本，无结构化评分

**需要开发**:

  - 时间复杂度标注 (Big O)
  - 空间复杂度标注 (Big O)
  - 代码风格评分 (0-10)
  - 具体改进建议列表
- [ ] **response_formatter.py**: 实现 `OPT-006`，将LLM输出解析为结构化格式
- [ ] **前端代码审查结果卡片**: 不同维度的评分可视化展示

**验收标准**:
- 代码审查响应包含可解析的结构化字段
- 前端能区分展示"正确性/复杂度/风格"三个维度

## 四、工程技术缺失（Spec中的OPT项）

### 4.1 P0 — 影响核心体验

| ID | 问题 | 描述 | 影响 |
|----|------|------|------|
| OPT-002 | Workers用complete()而非stream() | 用户等待全量生成完才看到内容，感知延迟高 | 首字响应延迟达5-15秒 |
| OPT-003 | _load_context()重复定义 | main.py中第117行和第123行重复定义，后者覆盖前者 | 潜在bug |

### 4.2 P1 — 安全性与完整性

| ID | 问题 | 描述 | 影响 |
|----|------|------|------|
| OPT-004 | 无Scope Guard | 非编程问题也发给LLM | 浪费token，无边界控制 |
| OPT-005 | 无Boundary Router | 意图路由粗粒度 | 无法区分"要答案"vs"要提示" |
| OPT-006 | 无Response Formatter | LLM输出纯文本 | 审查结果无结构化评分 |
| OPT-012 | 无LLM耗时追踪 | metrics Histogram存在但未observe | 无法测量生产环境延迟 |

### 4.3 P2 — 健壮性

| ID | 问题 | 描述 | 影响 |
|----|------|------|------|
| OPT-008 | StateManager无并发保护 | active_states是普通dict | 多WS写入同session可能丢更新 |
| OPT-010 | 无Web检索 | 无法查文档/版本差异 | 回答仅靠训练数据 |
| OPT-011 | 前端冗余Agent检测 | regex检测已被agent_info事件取代 | 死代码，可能误判 |
| OPT-013 | 无用户级速率限制 | 任意用户无限请求 | 滥用风险 |

## 五、视频录制所需功能准备

### 5.1 管理员视频（导入新题目）

**需展示流程**:
1. 登录OJ Admin后台 → Problem管理
2. 创建新题目（手动填写题面、输入输出、时间/内存限制）
3. 上传测试数据（test_case ZIP）
4. 设置题目标签与难度
5. 设为可见发布
6. 触发AI审计检查模板质量（`POST /audit/run/{problem_id}`）
7. 验证题目可在前端浏览

**当前可用**: 步骤1-7全部可实现，OJ Admin + AI审计均在线

**需提前准备**:
- [ ] 准备1-2道新题目的完整数据（题面MD + 测试数据ZIP）
- [ ] 确保Admin后台可正常访问操作

### 5.2 普通用户视频（系统练习）

**需展示流程**:
1. 注册/登录
2. 浏览题目列表（按难度/标签筛选）
3. 选中一题 → 查看题面
4. AI聊天中讨论题意（ProblemAnalyzerAgent）
5. 写代码 → 提交 → 查看判题结果
6. 代码有误 → AI聊天粘贴代码寻求帮助（CodeReviewerAgent）
7. 表达困惑 → AI学习伙伴回应（LearningPartnerAgent）
8. 询问学习规划 → 学习管理建议（LearningManagerAgent）
9. 询问竞赛策略 → 竞赛教练建议（ContestCoachAgent）

**当前可用**: 步骤1-8均可实现，步骤9有限（无真实竞赛数据）

**需提前准备**:
- [ ] 创建5-10个测试学生账号
- [ ] 确保至少50题有模板和测试数据
- [ ] 准备一段代码用于演示代码审查
